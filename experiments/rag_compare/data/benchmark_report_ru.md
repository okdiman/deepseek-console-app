### В2 — Scaled dot-product attention в трансформерах
**→ НИЧЬЯ** | Источник найден: ✓ `transformer_architecture.md`

| | Без RAG | С RAG |
|--|---------|-------|
| Ключевые слова | 4/4 (100%) | 4/4 (100%) |
| Время | 7.9с | 7.0с |

**Без RAG:**
> Scaled dot-product attention вычисляет оценки выравнивания между query и key, масштабирует их для стабильности градиентов, применяет softmax для весов и суммирует соответствующие value. Формула: Attention(Q, K, V) = softmax(QK^T / √dk) · V

**С RAG:**
> Механизм обрабатывает отношения между всеми элементами последовательности одновременно. Масштабирование — деление произведения query и key на корень из размерности ключа — предотвращает слишком большие значения перед softmax.

**Вывод:** Ответ без RAG полнее — включает формулу. RAG-ответ осторожнее ("детали не указаны в документации"). Оба попали по всем словам.

### В5 — Threading vs multiprocessing в Python
**→ БЕЗ RAG ↑** | Источник найден: ✓ `python_concurrency_guide.md`

| | Без RAG | С RAG |
|--|---------|-------|
| Ключевые слова | **3/3 (100%)** | 1/3 (33%) |
| Время | 6.7с | 5.7с |

**Без RAG:**
> **Threading**: потоки в одном процессе, общая память, ограничен **GIL** (**Global Interpreter Lock**), подходит для I/O-задач. **Multiprocessing**: отдельные **процессы**, своя память, обходит GIL, подходит для CPU-задач.

**С RAG:**
> Threading — потоки внутри одного процесса. Multiprocessing — отдельные процессы с собственной памятью. *(GIL не упомянут)*

**Вывод:** Найденный чанк описывал конкурентность концептуально, без секции про GIL. Классика проблемы "правильный документ, неправильный чанк".

## Анализ

### Где RAG побеждает: проектная специфика (В7, В8, В9)

Вопросы про архитектуру проекта — хуки, FSM, форматы расписаний — это зона RAG. У модели нет обучающих данных по этим темам. Без RAG ответы общие и неверные по контексту. С RAG — точные, со ссылками на конкретные классы и методы.

### Где Plain побеждает: общеизвестные темы (В3, В4, В5, В6)

Для тем из обучающих данных модели (теория RAG, LLM, threading, FastAPI) параметрические знания сильнее найденных чанков. Два паттерна отказа:

1. **Найден не тот документ** (В3): эмбеддинг привёл к `fastapi_overview.md` вместо `retrieval_augmented_generation.md`. Модель не стала галлюцинировать — отказалась отвечать. Хорошее поведение, но бесполезный результат.
2. **Правильный документ, неправильный чанк** (В4, В5): нужный файл попал в топ-3, но конкретная секция (про ICL, про GIL) оказалась в другом чанке.

### Точность источников: 8/10

Два промаха:
- **В3**: полный сбой поиска — совсем другая тематика документов
- **В9**: `scheduler_store.py` не найден, но `CLAUDE.md` (который тоже содержит нужное) нашёлся — функционально правильно, формально промах

### Время ответа

RAG добавляет 1–5с на эмбеддинг и поиск. На проектных вопросах (В7, В8) RAG-ответы были значительно длиннее и детальнее — отсюда 11–13с против 5–8с без RAG.

## Выводы и рекомендации

**RAG незаменим для проектной специфики.** Без него модель описывает планировщик ОС вместо FSM проекта и придумывает абстрактные хуки вместо конкретного `AgentHook`.

**На общих знаниях RAG нейтрален или мешает** при плохом ретривере. Модель ведёт себя честно — не галлюцинирует при нерелевантном контексте, но это означает пустой ответ.

**Узкое место — качество ретривера**, не LLM.

| Проблема | Решение |
|----------|---------|
| В3: найден не тот документ | Гибридный поиск (BM25 + embedding) — keyword "retrieval-augmented" точно найдёт нужный файл |
| В4, В5: нужный документ, не тот чанк | Увеличить `top_k` до 5 или добавить re-ranking |
| В10: ключевые слова в деталях | Добавить `mcp_manager.py` в корпус (сейчас его нет в `CORPUS_FILES`) |
